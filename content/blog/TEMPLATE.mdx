This is the lede paragraph. It introduces the post in two or three sentences,
sets context, and tells the reader why they should keep going. The lede
inherits the prose typography — no styling decisions are made here.

## A section heading uses h2

Body paragraphs flow underneath. Successive headings get vertical breathing
room from the prose mixin. Inline `code` looks like this — monospace, a
faint background tint, and a small radius.

> Blockquotes are indented with a left border. Italicize them only if the
> source itself was emphasized; otherwise let the border and color carry
> the visual weight.

### A subsection uses h3

Fenced code blocks always declare a language. Without a language hint,
Shiki can't tokenize and the block falls back to plain text.

```ts
type Post = {
  title: string;
  date: string;
  description?: string;
  tags?: string[];
  slug: string;
};

export function compareByDate(a: Post, b: Post): number {
  return b.date.localeCompare(a.date);
}
```

Lean snippets render the same way, even though Lean syntax is uncommon:

```lean
theorem add_comm (a b : Nat) : a + b = b + a := by
  induction a with
  | zero      => simp
  | succ a ih => simp [Nat.succ_add, ih]
```

Tables work for tabular data — never for layout.

| Column           | Purpose                                       |
| ---------------- | --------------------------------------------- |
| `title`          | Display name, shown in the index and the `<h1>` |
| `date`           | ISO-8601 date string, used for sorting        |
| `tags`           | Optional array of short, lowercase strings    |

Lists are unstyled by default — they inherit the prose font and spacing.

- One item per line.
- Keep items short; if an item needs a paragraph, restructure into prose.
- Use ordered lists only when sequence matters.

That's the full surface a post is expected to use. If something here looks
broken, the prose CSS in `_prose.scss` or the article shell needs to
absorb the fix — never patch it at the post level.
