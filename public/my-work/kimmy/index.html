Why this discipline

The wrong-spec problem

The artifact you built is correct against the thing you wrote down — and the thing you wrote down is wrong against reality.

Humans aren't precise about what they want. LLMs are literal executors. The outputs get more convincing as models improve, even when wrong. Divergence between what was asked for and what was built compounds silently until trust breaks.

"The spec is the bug." Calling the resulting issue a bug in the code misdirects attention to a layer where there is nothing to fix.

Method · the discipline

Every claim formalized before code

kimmy's source code is downstream of a Prolog claim that has already survived adversarial disprove rounds. The union of verdict/2 facts is the operative spec.

Code that breaks a claim breaks the spec build before the test suite ever runs.

Method · adversarial

LLM contradictions caught before code shipped

Between stages 2 and 3, every labeled claim passes through an adversarial disprove round. The pipeline tries to refute itself before any source file exists — and that's where the LLM's own spec-drift surfaces.

49disprove rounds on disk 8LLM contradictions / spec-drift events caught 27+downstream bugs pre-empted

Each refutation is an LLM contradiction or spec-drift event — the model proposed code or a claim that disagreed with the substrate's existing facts. Catching that contradiction before implementation is where the throughput comes from. Eight catches; twenty-seven downstream bugs prevented.

An unattacked specification is a guess that has not been challenged. A specification that has withstood a serious attempt at refutation is a guess with evidence behind it.

B-util · 1 / 5

Three-axis utilization

Utilization isn't a single number. kimmy reports a billable axis plus two productive denominators, leaving room for the user to audit.

  • Billable axis hours that bill to a client, against the contracted target
  • Productive denominator A billable + internal-tooling work, vs. capacity
  • Productive denominator B billable + tooling + meta (process, planning, learning)

A single number lies more confidently than three. The user gets to see which one they care about.

B-util · 2 / 5

Three gap-numbers, per axis

  • Catch-up to pace hours needed to be on pace for the period target as of today
  • Hours to period target total hours remaining to hit target by period end
  • Projected end-of-period straight-line extrapolation from current rate

All three over nine period kinds including anniversary_year — annual reviews land on hire-date anniversaries, not Jan 1.

B-util · 3 / 5

Why three axes? ADR-006.

The canonical utilization metric lives in Looker. The disprove round established that any single number kimmy reports will diverge from Looker for some user — different denominators, different rounding, different in-flight handling.

kimmy reports the inputs and lets the user audit, rather than report a number that disagrees with the dashboard.

The discipline isn't "give the right number." It's "make the wrongness visible."

B-util · 5 / 5

Whose client? Not the spec's.

KIM-035 — the multi-tenant identity invariant. No real person or client identity may appear anywhere in the committed tree.

  • The spec names no one no operator id, no employer, no customer. The only identities that survive in git are synthetic stand-ins — Alex Kim · alex.kim@example.com, Example Client, Example Billable Project, activity Development.
  • Real classification is user-local the actual customer / project / activity tuples live in ~/.config/kimmy/customer-projects.toml — read at runtime, never committed. Each operator's billable world is their own.
  • Enforced by a strict gate check.sh runs check_identity_invariant.sh --strict against a registry in spec_kb/identity_policy.pl. Any forbidden token in the tree fails the build (exit 4). The invariant is self-enforcing, not a convention.
Single operator today, US + Paraguay tomorrow. The spec was de-principaled before the rollout, not after — so the gate, not a reviewer, keeps identity out of the substrate.

B-parser · 2 / 4

Idle-gap collapsing

Default 5-minute threshold; the original guess was 15 minutes.

Disprove round 3 surfaced the failure mode: 15 minutes was wide enough that a multi-minute network hiccup or tool-call wait would inflate the previous block, double-counting the idle stretch as work.

The 300-second value isn't arbitrary; it's what the refutation drove. The spec records why.

B-parser · 3 / 4

The DST handling came from a disprove round that constructed three fall-back-day witnesses (1:30am occurring twice) and broke the naive date-key. Fold-tuple key restored uniqueness.

B-wrongspec · 1 / 5

What it is

"The artifact you built is correct against the thing you wrote down, and the thing you wrote down is wrong against reality."

This is the location-of-the-bug problem. It is not inside the implementation. It is one level up, in the document the implementation faithfully obeys. The same shape appears under different names across fields:

Statistics specification error model fit was correct, key assumption wrong
AI alignment outer misalignment system optimized stated objective faithfully, objective was a wrong proxy
Engineering / construction wrong dimension on the drawing build faithful, drawing wrong
B-wrongspec · 2 / 5

Why it's getting worse

For roughly fifteen years, downstream engineers silently absorbed a great deal of the wrong-spec problem. They asked the right question, raised the right flag, filled the gap with judgment on the way down. The spec was the starting point; humans negotiated the rest.

AI agents are literal executors. They do not pause to ask whether the thing you asked for is the thing you wanted. They build what was written, and they build it quickly. Gaps which used to get caught in human-to-human handoff now ship to production before anyone has looked at them.

The cost of a requirements error grows superlinearly with how late it is caught. NASA estimates put the ratio at roughly 1× at spec stage, up to 1,500× after release.

B-wrongspec · 3 / 5

The diagnosis

"The spec is the bug." The implementation built exactly what was written, and that is the problem, not its solution.

Calling the resulting issue a bug in the code misdirects attention to a layer where there is nothing to fix. A bug-in-code mindset hunts for defects in the implementation. A bug-in-spec mindset hunts for defects in the assumptions, vocabulary, omissions, and frame of the document.

B-wrongspec · 4 / 5

The common toolkit

Every standard mitigation is, underneath, the same move: treat the spec as a hypothesis, not a contract.

  • Prototype before specifying collide the spec with reality in hours
  • Specification by example force prose ambiguity into Given/When/Then
  • Working backwards write the press release before building
  • Riskiest-assumption testing rank assumptions; test the top one
  • Tight iteration loops ship a vertical slice; observe
  • Pre-mortems "imagine this shipped and users hated it — why?"

Push the cost of being wrong as far left as possible. Every technique is a way of moving the moment of contradiction earlier on the cost curve.

B-wrongspec · 5 / 5

The formal turn — four layers of defense

Where the domain admits it, formal methods strengthen the toolkit. The strongest available answer composes four layers:

4Adversarial conceptual reviewcatches frame errors
3Counterfactual constraintscatches spec omissions
2Shared ontologycatches semantic mismatch
1Two-system cross-checkcatches implementation drift
  • 1 · Two-system cross-check Prolog from code, Lean from spec, properties checked for agreement. Catches implementation drift.
  • 2 · Shared ontology both systems draw from a common axiomatized vocabulary. Catches semantic mismatch.
  • 3 · Counterfactual constraints positive + negative condition for every property. Catches spec omissions.
  • 4 · Adversarial conceptual review external party attacks the spec at the frame level. Catches frame errors.

B-pipeline · 2 / 4

Stages 1–3 · grounding intent in substrate

  • close-world CWA over source. Whatever isn't in existing-world.pl is, by assumption, false. Becomes the baseline.
  • decompose-proposition walks the proposition through a counterfactual lens. Emits hypothesis.pl: labeled claims (descriptive / counterfactual / prescriptive) + query evidence + formal-property sketches.
  • disprove (between 2 and 3) adversarial round — try to refute each claim before any source exists. Counterexamples that survive land in counterexamples.pl.
  • model-obligations emits target-world.pl — the substrate Lean later proves against — plus per-property verdicts.
B-pipeline · 3 / 4

Stages 4–5 · proof + instantiation

  • prove-invariants reads target-world.pl, translates each formal property into a Lean 4 theorem with mandatory provenance annotation, proves via lean-spec-writer + lean-expert sub-agents. Emits lean/Proofs/*.lean + lean_proof_results.pl.
  • instantiate-properties reads the Lean proofs, instantiates each universal property as a projection test at a specific fixture, or emits a behavioral_claim test for I/O / state / concurrency / timing. All tests start skipped.

Universality is intentionally discarded at the lean → tdd boundary. The Lean theorem says "for all inputs x, P(x)"; the test pins one specific x and verifies P holds there. The test isn't the proof — the proof was already done.

B-pipeline · 4 / 4

Stages 6–7 · realize + measure

  • realize-specification unskips one test at a time, routes by test_category (projection vs behavioral_claim) and by ontology label of the cited claim. Modifies source under the target codebase directory; emits implementation_log.md.
  • measure-entailment extracts claims from each input resource as Prolog facts, then scores overlap, gaps, contradictions, extensions. In pipeline-terminal mode runs label-aware verdicts: Pattern 3 (counterfactual still present), prescriptive fulfillment, prescriptive negation violations.

The pipeline is bookended by adherence checks — close-world says what's true now, measure-entailment says whether the implementation actually entails the original proposition.

B-disprove · 1 / 1

The mechanic

For each labeled claim in hypothesis.pl, the adversary attempts to refute the claim within a fixed budget. Three possible verdicts:

  • refuted a witness is deposited to counterexamples.pl. The claim is wrong or under-specified. Caller must revisit.
  • inconclusive partial evidence found; recorded for review. Doesn't fail the build.
  • abstained no progress within budget; reason recorded. The dialog itself is the evidence — recorded even on abstain per spec-first rule #2.

Counterexamples are first-class artifacts. They carry forward through every subsequent pipeline stage as constraints that the spec must respect. A refuted verdict is what the headline numbers report as an "LLM contradiction caught."

Frontier-lab paper on CoT reasoning with separate-verifier scaffolding. Read the PDF →

  • Mechanism labeled-claim decomposition (descriptive / counterfactual / prescriptive) with adversarial disprove round before code generation. WybeCoder proves-as-you-generate; orbital attacks the spec first, then generates against the survivors.
  • Surface seven discrete pipeline stages with provenance threading between them. Each stage is independently runnable; partial re-runs are first-class. The pipeline is a formalized debate, not a verdict machine.

The convergence is the point. The differentiation is the proof that the architecture has been thought through.
